# /verify-loop

> /verify의 확장판. 검증 실패 시 자동 수정 후 재검증을 반복 (최대 5회).

## 사용법

```
/verify-loop
/verify-loop --max-retries <N>
/verify-loop --only build|test|lint
```

- `--max-retries`: 최대 재시도 횟수 (기본: 5)
- `--only`: 특정 검증만 실행

## 실행 절차

### Step 1: 환경 파악

1. 프로젝트 타입 감지 (빌드 명령어, 테스트 프레임워크 등)
2. 관련 설정 파일 확인
3. 검증 대상 결정 (build, test, lint)

### Step 2: 코드 리뷰 (자동)

verify-agent 스폰 전, 변경된 코드를 빠르게 리뷰:
- 로직 오류 검사
- 엣지 케이스 누락 확인
- 보안 취약점 확인
- 명백한 문제 발견 시 선제적 수정

### Step 3: 검증 루프

```
시도 1/5:
  빌드... ✅
  테스트... ❌ 2 failures
  → 자동 수정: import 경로 수정, 미사용 변수 제거

시도 2/5:
  빌드... ✅
  테스트... ✅
  린트... ✅
  → 전체 통과!
```

### 자동 수정 가능 에러

| 유형 | 예시 | 수정 방법 |
|------|------|----------|
| Import | 잘못된 경로, 누락된 import | 경로 수정, import 추가 |
| Lint format | 들여쓰기, 세미콜론 | 포매터 실행 |
| Unused | 미사용 변수, import | 제거 |
| Simple type | 누락된 속성, 타입 불일치 | 타입 수정 |

### 수동 수정 필요 에러

자동 수정 불가 시 가이드 제공:
- 비즈니스 로직 에러
- 설계 변경 필요한 타입 에러
- 테스트 실패 (구현 수정 필요)

### Step 4: 결과 보고

#### 성공 시
```
## /verify-loop 완료 ✅

- 시도: 2/5
- 빌드: ✅
- 테스트: 15 passed
- 린트: ✅
- 자동 수정: 2건 (import 1, unused 1)

### 다음 단계
- /commit: 커밋
```

#### 실패 시
```
## /verify-loop 실패 ❌

- 시도: 5/5 (최대 도달)
- 반복 에러: src/auth.ts:45 — 타입 불일치 (5회 반복)

### 수동 수정 필요
1. src/auth.ts:45 — `User` 타입에 `role` 속성 추가 필요
   힌트: UserRole enum 정의 후 User 인터페이스에 추가

### 다음 단계
- 수동 수정 후 /verify-loop 재실행
```

## /verify와의 차이

| 항목 | /verify | /verify-loop |
|------|---------|-------------|
| 재시도 | 내장 루프 (최대 5회) | 명시적 루프 (기본 5회) |
| 자동 수정 | Fixable 에러 자동 수정 | 동일 + 수정 불가 에러 가이드 제공 |
| 코드 리뷰 | 없음 | 검증 전 자동 코드 리뷰 포함 |
| 실패 보고 | 단순 FAIL | 반복 에러 추적 + 수동 수정 힌트 |

## 핵심 규칙

- 자동 수정은 Import, Lint format, Unused, Simple type만 대상
- 비즈니스 로직 에러는 자동 수정하지 않고 가이드만 제공
- 동일 에러가 반복되면 수동 수정 필요로 분류
- 검증 전 코드 리뷰로 명백한 문제를 선제적 수정
